# Гипотезы по оптимизации RAG Pipeline

## 1. Связи всех модулей

### Pipeline Flow
```
questions.csv → kaggle_main.py → Retriever → Generator → submission.csv
                    ↓              ↓         ↓
              config.py      retriever.py  generator.py
                    ↓              ↓         ↓
              chunker.py       indexer.py    (fallback)
                    ↓              ↓
              indexer.py       (FAISS + BM25 + Reranker)
```

### Модуль `config.py`
- **Что делает**: Централизованная конфигурация
- **Ключевые параметры**:
  - `TOP_K_RETRIEVAL=40` - кандидаты из FAISS
  - `TOP_K_BM25=15` - кандидаты из BM25
  - `TOP_K_RERANK=10` - финальные результаты
  - `RERANKER_BATCH_SIZE=15` - батч для reranker
  - `MAX_SENTENCES=2` - максимум предложений в ответе
  - `MAX_RESPONSE_CHARS=150` - жёсткий лимит символов

### Модуль `chunker.py`
- **Что делает**: Разбивает сайты на чанки
- **Параметры**:
  - `CHUNK_SIZE=450` - размер чанка в символах
  - `CHUNK_OVERLAP=100` - перекрытие
- **Влияние на скор**: Большие чанки = больше контекста, но медленнее

### Модуль `indexer.py`
- **Что делает**: Создаёт FAISS индекс
- **Модели**:
  - `EMBEDDER_MODEL=BAAI/bge-m3` (1024-мерные эмбеддинги)
- **Время**: ~1-2 минуты на индексацию

### Модуль `retriever.py`
- **Что делает**: Гибридный поиск
- **Pipeline**:
  1. FAISS semantic search (TOP_K_RETRIEVAL кандидатов)
  2. BM25 lexical search (TOP_K_BM25 кандидатов)
  3. Merge + deduplicate
  4. Cross-encoder reranking (TOP_K_RERANK финальных)
- **Модели**:
  - `RERANKER_MODEL=BAAI/bge-reranker-v2-m3`
- **Время**: ~1-2 секунды на запрос

### Модуль `generator.py`
- **Что делает**: Генерация ответа
- **Pipeline**:
  1. LLM generation (если context не пуст)
  2. Fallback: `extract_answer_from_context()` (TF-IDF + позиция + информативность)
- **Время**: ~3-4 секунды на запрос (Vikhr-1B)

### Модуль `kaggle_main.py`
- **Что делает**: Оркестрация
- **Pipeline**:
  1. Build/load index
  2. Create retriever
  3. For each question: get_context() → generate() → validate()
- **Default модель**: `vikhr-1b-finetuned`

## 2. Что можно менять (без риска)

### Параметры, которые НЕ влияют на время
- `MAX_SENTENCES` (1-3) - количество предложений в ответе
- `MAX_RESPONSE_CHARS` (100-200) - обрезка ответа
- `TEMPERATURE` (0.0-0.3) - детерминированность

### Параметры, которые можно увеличить (риск: время)
- `TOP_K_RERANK` (5-15) - **безопасно**, увеличивает recall
- `RERANKER_BATCH_SIZE` (10-20) - **безопасно**, ускоряет reranking

### Параметры, которые можно уменьшить (риск: скор)
- `TOP_K_RETRIEVAL` (15-25) - уменьшит recall, ускорит
- `TOP_K_BM25` (10-15) - уменьшит recall, ускорит

---

## 3. Что НЕЛЬЗЯ менять (риск: время > 12 часов)

### Критические лимиты
- **FAISS + BM25 > 50 кандидатов** - reranking займет >2 сек/запрос
- **RERANKER_BATCH_SIZE > 20** - CUDA OOM на Kaggle T4
- **LLM > 7B** - генерация >5 сек/запрос

### Текущие настройки (баланс)
- `TOP_K_RETRIEVAL=40` + `TOP_K_BM25=15` = ~50 кандидатов
- После merge: ~40-45 уникальных
- Reranking: 40/15 = 3 батча × 0.5 сек = ~1.5 сек
- Генерация: ~4 сек
- **Итого: ~5-6 сек/запрос** ✓

## 4. Гипотезы по улучшению скора

### Гипотеза 1: Увеличить TOP_K_RERANK
- **Сейчас**: 10
- **Можно**: 12-15
- **Ожидаемый эффект**: +0.02-0.05 балла
- **Риск**: +0.5-1 сек/запрос

### Гипотеза 2: Увеличить RERANKER_BATCH_SIZE
- **Сейчас**: 15
- **Можно**: 18-20
- **Ожидаемый эффект**: -0.5 сек (ускорение)
- **Риск**: CUDA OOM

### Гипотеза 3: Изменить порядок чанков
- **Сейчас**: reversed (Lost in the Middle fix)
- **Можно**: не реверсить, пробовать разные варианты
- **Ожидаемый эффект**: +0.01-0.03 балла

### Гипотеза 4: Длиннее контекст
- **Сейчас**: 10 чанков × ~300 симв = ~3000 симв
- **Можно**: увеличить TOP_K_RERANK до 15
- **Ожидаемый эффект**: +0.02-0.05 балла
- **Риск**: время

### Гипотеза 5: SYSTEM_PROMPT для Vikhr
- **Сейчас**: общий русскоязычный
- **Можно**: добавить специфичные для Alfa-Bank примеры
- **Ожидаемый эффект**: +0.01-0.03 балла

## 6. Рекомендации

### Безопасные изменения:
1. `TOP_K_RERANK=12` - чуть больше контекста
2. `RERANKER_BATCH_SIZE=18` - чуть быстрее

### Опасные изменения:
1. `TOP_K_RETRIEVAL > 40` - время > 12 часов
2. `TOP_K_BM25 > 20` - время > 12 часов
3. Любой 7B модель - время > 12 часов

### Альтернатива:
- Использовать `OR_main.py` (OpenRouter API) - ~5-6 сек, но интернет
